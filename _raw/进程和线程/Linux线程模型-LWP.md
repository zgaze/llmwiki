### Linux线程模型

* linux 线程模型是1:1的，每个线程都对应内核中的一个调度实体内核级线程。
* linux 2.6以后，使用新的pthread线程库–NPTL(Native POSIX Threading Library)，是一种LWP的实现机制
* linux的线程就是轻量级进程，所有的线程都当作进程来实现，因此线程和进程都是用task\_struct来描述的。
* LWP的操作，如建立、析构以及同步，都需要进行系统调用。系统调用的代价相对较高：需要在user mode和kernel mode中切换。其次，每个LWP都需要有一个内核线程支持，因此LWP要消耗内核资源（内核线程的栈空间）。
* 对于一个进程，相当于是它含有一个线程，就是它自身。对于多线程来说，原本的进程称为主线程，它们在一起组成一个线程组。
* 进程拥有自己的地址空间，所以每个进程都有自己的页表。而线程却没有，只能和其它线程共享某一个地址空间和同一份页表。在进程/线程创建时，因是否拷贝当前进程的地址空间还是共享当前进程的地址空间，而使得指定的参数不同而导致的。具体地说，进程和线程的创建都是执行 clone 系统调用进行的。而 clone 系统调用会执行 do\_fork 内核函数，而它则又会调用 copy\_process 内核函数来完成。
* 创建线程copy\_process主要包括如下操作：
	* 在调用 copy\_process 的过程中，会创建并拷贝当前进程的 task\_stuct，同时还会创建属于子进程的 thread\_info 结构以及内核栈。
	* 此后，会为创建好的 task\_stuct 指定一个新的 pid（在 task\_struct 结构体中）。
	* 然后根据传递给 clone 的参数标志，来选择拷贝还是共享打开的文件，文件系统信息，信号处理函数，进程地址空间等。这就是进程和线程不一样地方的本质所在。

#### 内核线程

* 内核线程运行在系统栈(内核栈)，内核线程只运行在内核态，普通进程既可以运行在内核态，也可以运行在用户态。
* 内核线程只使用 3-4GB (假设为 32 位系统) 的内核地址空间（共享的），即高位内存，但普通进程由于既可以运行在用户态，又可以运行在内核态，因此可以使用 4GB 的虚拟地址空间。
* 内核进程的active\_mm，就是交出cpu使用权的进程的struct\_mm，这个用来延迟刷新TLB，甚至不刷新。
* 内核线程的ppid是2，启动时候创建的。

#### 非1:1线程模型

* 用户级线程的切换显然要比内核级线程的切换快一些, 前者可能只是一个简单的长跳转, 而后者则需要保存/装载寄存器, 进入然后退出内核态. (进程切换则还需要切换地址空间等.)
* 用户级线程则不能享受多处理器, 因为多个用户级线程对应到一个内核级线程上, 一个内核级线程在同一时刻只能运行在一个处理器上.不过, M:N的线程模型毕竟提供了这样一种手段, 可以让不需要并行执行的线程运行在一个内核级线程对应的若干个用户级线程上, 可以节省它们的切换开销.

#### 创建新线程过程

分配栈空间，初始化ptrhead结构体 pthread\_create对线程的创建实际上是向pthread\_manager\_thread发送个request请求
pthread\_create { struct pthread\_request request; request.req\_thread = thread\_self(); request.req\_kind = REQ\_CREATE; request.req\_args.create.attr = attr; request.req\_args.create.fn = start\_routine; request.req\_args.create.arg = arg; _// 向\_\_pthread\_manager\_request写入request请求。 request请求将被\_pthread\_manager\_thread读取并处理。_ _// 管理线程与其他线程之间的通信是通过管道完成的。_ _// 在一个进程空间内，管理线程与其他线程之间通过一对"管理管道（manager\_pipe\[2\]）"来通讯，_ \_\_libc\_write(\_\_pthread\_manager\_request, (char \*) &request, sizeof(request)); _// 挂起一下_ suspend(self); _// 线程添加到\_\_pthread\_main\_thread 双向链表中 由管理线程管理_ }

* 新建的线程的栈在主线程的栈顶下面（即地址小于主线程的栈顶），创建线程的时候，首先计算新线程的栈地址，然后调用mmap划出这块地址。否则访问的时候会segmentfault。最后调用clone创建进程（线程）并设置栈的栈顶和栈底位置。
* 管理线程作为pthread\_create()的调用者线程的子线程运行，而pthread\_create()所创建的那个用户线程则是由管理线程来调用clone()创建，因此实际上是管理线程的子线程。（此处子线程的概念应该当作子进程来理解。）-- 也就是说 创造出的线程实际都是管理线程的自线程（子进程）
* 在LinuxThreads中，专门为每一个进程构造了一个管理线程，负责处理线程相关的管理工作。当进程第一次调用pthread\_create()创建一个线程的时候就会创建（\_\_clone()）并启动管理线程。 linux进程和线程都是用sys\_clone系统调用完成的。本质上说，每个线程其实就是一个进程，只不过线程共享部分资源。CLONE\_VM（共享内存空间）、CLONE\_FS（共享文件系统信息）、CLONE\_FILES（共享文件描述符表）、CLONE\_SIGHAND（共享信号句柄表）
* 线程对操作系统来说就是进程，因此对信号的处理是以进程为单位分发的；线程总数受限于系统进程数； 线程之间的调度是由内核调度器来处理的

#### 进程切换场景

进程切换一定发生在中断／异常／系统调用处理过程中，常见的有以下情况：

* 阻塞式系统调用、虚拟地址异常，导致被中断进程进入等待态。
* 时间片中断、I/O中断后发现更改优先级进程，导致被中断进程进入就绪态。
* 终止用系统调用、不能继续执行的异常，导致被中断进程进入终止态。

#### 线程切换过程

* 触发中断，可能是时间片用完，或者进入阻塞式系统调用，或者线程终止。
* CPU暂停执行下一条要执行的指令转而去执行与中断信号对应的处理程序，（用户栈保存pc cs：ip寄存器组，eflags）这样就完成了用户态到内核态的切换。
* 切换到内核堆栈（task\_struct->stack字段）。然后saveall（包括各种寄存器）
* ret返回后，取出各个寄存器的值。
* 判断是否需要调度；需要的话，会执行schedule函数
* schedule()用户线程没法调度，只有进入内核态才能执行
* schedule——>\_\_schedule——>pick\_next\_task():pick\_next\_task这个函数封装了进程调度算法，他返回下一个需要调度的进程。
* 找到需要调度的函数后，通过调用 context\_switch(保存sp，ip（pc指令寄存器，换成新进程的sp，ip)——>switch\_to进行切换\`:
* 线程切换的时候，用户态的栈信息保存在task\_struct中的成员thread中.

#### 线程切换开销比进程小的原因

* CPU上运行了若干的用户空间的进程和内核线程，为了加快性能，CPU中往往设计了TLB和Cache这样的HW block。Cache为了更快的访问main memory中的数据和指令，而TLB是为了更快的进行地址翻译而将部分的页表内容缓存到了Translation lookasid buffer中，避免了从main memory访问页表的过程。 因此，B进程刚开始执行的时候，TLB miss和Cache miss都非常严重，从而导致了性能的下降。
* 线程切换的时候，内核线程的active\_mm指向上一个线程的用户空间，如果是同进程的话，一直就不需要刷TLB，因此线程切换的开销比进程小。
